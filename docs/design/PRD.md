# Jarvis 产品需求文档

版本：v2.0
状态：MVP 需求基线（CA Agent + Worker 架构）
来源：由 v1.0 演进，引入上下文感知主脑与并行 Worker 执行层

## 1. 产品定位

Jarvis 是一个常驻本地运行的个人 AI 助理。它通过飞书接收自然语言指令，使用**上下文感知主脑（CA Agent）**理解目标、管理上下文、调度任务，通过**并行 Worker 执行层**调用命令行、代码 Agent、GitHub 和 Obsidian，并在高风险操作前请求人工授权。

Jarvis 的目标不是提供普通问答，而是把用户的模糊需求转为**可并行执行、可验证、可恢复、可审计**的长期任务。

**架构定位**：

- **CA Agent**：负责高层逻辑、上下文管理、状态同步、任务分发。它是"大脑"，不直接执行 Skill。
- **Worker**：负责具体执行。Worker 无状态、无决策权，接收 `WorkOrder`、执行、返回 `WorkResult`。多个 Worker 可以并行运行。

## 2. 目标用户

- 个人开发者。
- 小团队技术负责人。
- 高频使用本地仓库、GitHub、命令行和 Obsidian 的工程用户。
- 需要把代码改造、调研、Review、归档等长耗时任务**异步并行委托**给 Agent 的用户。

## 3. 核心价值

- 降低开发者对长耗时任务的持续关注成本。
- 将自然语言指令拆解为**可并行执行的 WorkOrder 清单**。
- **Worker 并行执行**，提高任务吞吐量。
- 在本地环境完成代码修改、测试、总结和知识沉淀。
- 通过资源锁避免并发修改冲突。
- 通过人工授权拦截危险操作。
- 通过 checkpoint 支持断点恢复，通过业务 DB 补偿回调丢失。

## 4. MVP 范围

MVP 按阶段交付。Phase 1 先交付本地 CA Agent 事件循环和 Inline Worker，验证核心架构边界；Phase 2 再交付 Thread Worker 并行、资源锁和恢复补偿；飞书作为外部入口在 Phase 3 集成，不阻塞 Phase 1/2 的本地闭环。

### 4.1 完整 MVP 必须支持

1. 飞书文本消息接入（支持飞书机器人和个人机器人双通道）。
2. 飞书互动卡片授权回调。
3. **CA Agent 事件循环**：可多次 invoke、挂起、恢复。
4. **Worker 抽象层**：`WorkOrder`/`WorkResult` 接口，至少支持 Inline 模式。
5. **并行 Worker 执行**（Thread Worker 模式）：多个 Worker 可同时运行。
6. SQLite checkpoint。
7. 资源锁和任务会话映射（支持 Worker 锁继承）。
8. Shell Worker。
9. Obsidian 复盘写入（文件命名由 Agent 按需生成，每日工作日记采用 `YYYY-MM-DD 工作日记` 格式）。
10. 高危命令识别和拦截（**dispatch 前**动态判断风险等级）。
11. 任务状态查询（含 Worker 状态）。
12. 任务取消和资源释放。
13. Windows 本地运行（代码需保持对 macOS 的兼容性）。

### 4.2 MVP 暂不支持

1. 飞书语音转文字。
2. 截图识别和其他多模态输入。
3. 云端沙箱或多租户权限隔离。
4. 独立 Web Dashboard。
5. ChromaDB RAG。
6. GitHub PR 全自动创建。
7. 文件夹监听触发。
8. 多模型动态路由。
9. Worker 内部子图化（Subgraph Worker）。

## 5. 用户故事

### 5.1 异步代码开发

**作为** 开发者，
**我希望** 在飞书发送"帮我重构一下 IPOAgent 的登录逻辑"，
**以便** Jarvis 能自动分析、修改、验证并汇报结果。

验收标准：

- CA Agent 能识别目标仓库和任务类型。
- CA Agent 能生成 WorkOrder 清单。
- 每个 WorkOrder 都有 Worker 类型和 DoD。
- Worker 执行后返回结构化结果。
- CA Agent 聚合结果，判断目标是否达成。
- 最终飞书报告包含改动摘要、验证结果和失败项。

### 5.2 并行知识调研

**作为** 开发者，
**我希望** 在代码任务运行时继续发起调研任务，
**以便** Jarvis 不阻塞当前任务，同时能将调研结果归档到 Obsidian。

验收标准：

- 调研任务和代码任务由不同 Worker 并行执行。
- 调研任务不占用正在写入的代码仓库锁。
- 调研报告包含结论、来源、适用性分析和后续建议。
- 调研完成后写入 Obsidian 并通过飞书通知。
- CA Agent 在所有 Worker 完成后汇总汇报。

### 5.3 高危操作授权

**作为** 用户，
**我希望** Jarvis 在执行 `git push`、递归删除、生产部署等动作前请求授权，
**以便** 避免误操作造成损失。

验收标准：

- CA Agent 在 **dispatch Worker 前**识别高危 WorkOrder。
- 未授权前不得 dispatch Worker。
- 授权卡片展示命令、工作目录、风险等级和原因。
- 用户拒绝后任务进入 `blocked` 或 CA Agent 重新规划。
- 授权与拒绝都写入审计日志。

### 5.4 断点恢复与回调补偿

**作为** 用户，
**我希望** Jarvis 在进程崩溃或电脑重启后恢复任务，
**以便** 长任务不会因为环境中断而丢失，Worker 结果也不会因为回调丢失而永远等待。

验收标准：

- 进程启动后能发现未完成 CA thread。
- 可从最近一次成功 checkpoint 恢复。
- Worker 已完成但回调丢失时，系统能通过业务 DB 补偿，合成事件唤醒 CA Agent。
- 对已有外部副作用的动作不重复执行。
- 恢复后向飞书发送状态摘要。

### 5.5 Worker 超时与取消

**作为** 用户，
**我希望** 某个 Worker 卡住时能自动超时或被手动取消，
**以便** 不会无限等待，且 CA Agent 能根据超时信息重新决策。

验收标准：

- Worker 超过 `timeout_seconds` 后自动终止。
- CA Agent 收到超时结果后，可选择重试、换方案或放弃。
- 用户可手动取消正在运行的 Worker，释放资源锁。

## 7. 非功能需求

| 编号 | 需求 | 说明 | 验收标准 |
| --- | --- | --- | --- |
| N-1 | 资源隔离 | 同一物理目录写操作串行 | 并发请求不会同时修改同一仓库 |
| N-2 | 稳定性 | 单 Worker 失败不得导致主进程退出 | 关键异常可恢复 |
| N-3 | 可观测性 | 日志、状态、审计记录完备 | 可查询任务生命周期和 Worker 调用历史 |
| N-4 | 安全性 | 高危操作默认拦截 | 未授权不 dispatch 写远端、删除、部署 Worker |
| N-5 | 可配置性 | 飞书、仓库、模型、权限规则可配置 | 配置变更无需改代码 |
| N-6 | 上下文控制 | 长任务需要压缩摘要 | 任务历史可追溯且不会无限膨胀 |
| N-7 | 回调可靠性 | Worker 完成必须可靠通知 CA Agent | 支持业务 DB 补偿机制 |

## 8. 风险分级（动态判断）

风险等级由 CA Agent 在 **dispatch 前**结合上下文动态判断：

| 风险等级 | 判断依据 | 默认策略 |
| --- | --- | --- |
| Low | 只读查询、查看 Git 状态 | 直接 dispatch Worker |
| Medium | 运行测试、写 Obsidian 笔记、安装项目依赖 | 直接 dispatch，记录日志 |
| High | 修改代码、创建提交、推送分支、部署预览 | 根据配置决定是否生成 ApprovalRequest |
| Critical | 删除目录、强制推送、生产部署、修改全局配置 | **必须人工授权**后才 dispatch Worker |

**特殊规则**：所有 `git push` 操作无论上下文如何，默认必须人工授权。

## 9. 已确认决策

| 编号 | 问题 | 决策 |
| --- | --- | --- |
| 1 | 运行平台 | MVP 优先 Windows，代码保持跨平台兼容性 |
| 2 | 飞书接入 | 同时支持飞书机器人和个人机器人，通过配置切换 |
| 3 | Coder Worker 默认 | 封装为两个独立 Worker（Claude Code / Codex），用户通过配置选择默认 |
| 4 | Obsidian 命名 | 由 Agent 按需生成；每日工作日记采用 `YYYY-MM-DD 工作日记` 格式 |
| 5 | 任务恢复写操作 | 恢复后遇到写操作或 `git push` 必须先飞书通知用户确认，不得自动 dispatch Worker |
| 6 | 密钥管理 | 使用 `.env` 文件管理敏感配置，禁止提交到 GitHub |
| 7 | 风险分级方式 | CA Agent 在 dispatch 前动态判断，特殊规则：`git push` 一律需授权 |
| 8 | 架构模式 | **CA Agent + Worker 分离**，CA 负责决策，Worker 负责执行，Worker 可并行 |
| 9 | Worker 实现路径 | Phase 1 Inline（兼容）→ Phase 2 Thread（并行）→ Phase 3 Subgraph（自治） |
| 10 | LangGraph 版本 | 使用 1.0+ 特性（`interrupt()`、`Command`、`astream`） |
