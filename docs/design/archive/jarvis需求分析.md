# Long-run 极客 AI 助理需求分析报告

版本：v1.1  
状态：需求完善稿  
目标阶段：MVP / v1.0

## 1. 产品定位与愿景

Jarvis 是一个 7x24 小时常驻运行、具备任务闭环能力的个人 AI 助理。它以开发者本地电脑为宿主，以飞书为主要交互入口，以大模型为决策引擎，接管日常编码、知识调研、任务跟踪和结果归档工作。

产品目标不是“聊天机器人”，而是一个能长期运行、能记住上下文、能调用本地工具、能在关键风险点请求人工授权的任务执行系统。

## 2. 目标用户与使用边界

### 2.1 目标用户

- 个人开发者或小团队技术负责人。
- 高频使用本地仓库、GitHub、命令行和知识库的工程用户。
- 希望把代码改造、调研归档、例行检查等工作异步委托给 Agent 的用户。

### 2.2 核心价值

- 降低开发者对长耗时任务的持续关注成本。
- 把自然语言指令转为可验证的任务清单。
- 在本地环境完成代码修改、测试、总结和归档。
- 通过资源锁和人工授权控制并发冲突与高危操作。

### 2.3 v1.0 关键边界

- 第一阶段默认运行在可信本地机器上，不提供云端多租户能力。
- 第一阶段以文本指令为主，不处理语音、截图等多模态输入。
- 第一阶段不提供独立 Web Dashboard，状态通知收敛到飞书和 Obsidian。

## 3. 用户故事与核心场景

### 3.1 异步复杂代码开发

**故事**：作为开发者，我在飞书发送“帮我重构一下 IPOAgent 的登录逻辑”，Agent 能自动接管任务。

**预期行为**：

1. 解析用户意图并定位目标仓库。
2. 生成带完成定义（Definition of Done, DoD）的任务清单。
3. 调用本地代码 Agent 或 CLI 修改代码。
4. 执行测试、构建或静态检查。
5. 失败时读取错误并尝试修复。
6. 完成后通过飞书发送变更报告，并将复盘写入 Obsidian。

**验收标准**：

- 能识别仓库路径和任务类型。
- 每个子任务都有明确状态：`pending`、`running`、`success`、`failed`、`blocked`。
- 最终报告包含改动摘要、验证命令、验证结果和失败项。

### 3.2 并行知识调研

**故事**：当 Agent 正在重构代码时，我在飞书发送“帮我调研一下 LangGraph 的最新持久化方案”。

**预期行为**：

1. 判断该任务不占用当前代码仓库锁。
2. 创建独立会话执行调研。
3. 生成结构化调研报告。
4. 将报告写入 Obsidian。
5. 通过飞书通知调研完成。

**验收标准**：

- 调研任务不阻塞正在运行的代码任务。
- 调研报告包含来源、结论、适用性分析和后续建议。
- 对需要最新资料的问题必须进行联网检索或调用外部资料源，并记录来源。

### 3.3 高危操作人工授权

**故事**：Agent 准备执行 `git push origin main`、删除文件、部署生产环境等操作。

**预期行为**：

1. Action Router 判断操作风险等级。
2. 对高危操作保存当前状态并挂起。
3. 飞书发送授权卡片，展示命令、影响范围和建议。
4. 用户点击允许后继续执行，点击拒绝后终止该动作并记录原因。

**验收标准**：

- 未授权前绝不执行高危命令。
- 授权请求必须包含 `thread_id`、命令内容、工作目录、风险等级和超时策略。
- 拒绝后任务状态进入 `blocked` 或重新规划。

### 3.4 灾难恢复与断点续传

**故事**：本地机器重启或 Agent 主进程被杀死后，任务可以恢复。

**预期行为**：

1. 每个关键节点都写入 SQLite checkpoint。
2. 进程启动时扫描未完成任务。
3. 根据最后成功节点恢复执行。
4. 对执行中断的外部命令进行状态校验，避免重复破坏性操作。

**验收标准**：

- Agent 重启后能列出未完成任务。
- 可从最近一次成功 checkpoint 恢复。
- 恢复动作不会重复执行已成功的高危操作。

## 4. 功能需求

### 4.1 交互与触发模块

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| F-1.1 | 接收飞书单聊/群聊文本消息 | P0 | 能解析发送人、会话、文本、时间戳 |
| F-1.2 | 飞书互动卡片 | P0 | 支持允许、拒绝按钮及回调事件 |
| F-1.3 | 定时任务触发 | P1 | 支持 Cron 表达式或等价配置 |
| F-1.4 | 本地文件变化触发 | P2 | 能监听指定目录变化并产生内部事件 |
| F-1.5 | 状态查询指令 | P0 | 用户可查询当前任务、锁、失败项和待授权项 |

### 4.2 任务规划与执行引擎

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| F-2.1 | 任务拆解与 DoD | P0 | 每个子任务都有验证方式或人工验收说明 |
| F-2.2 | 闭环验证 | P0 | 执行后记录验证命令、退出码和输出摘要 |
| F-2.3 | 自我修正 | P0 | 验证失败后可重试，重试次数有上限 |
| F-2.4 | 任务暂停与恢复 | P0 | 支持人工授权、等待外部资源、异常阻塞 |
| F-2.5 | 任务取消 | P1 | 用户可取消未完成任务，并释放资源锁 |

### 4.3 本地技能调度系统

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| F-3.1 | 终端命令执行 | P0 | 支持工作目录、超时、输出截断、退出码 |
| F-3.2 | Claude Code 桥接 | P0 | 可作为 Skill 被 Planner 调用 |
| F-3.3 | GitHub 联动 | P1 | 支持查看 Issue、创建分支、提交 PR |
| F-3.4 | 命令风险分级 | P0 | 命令执行前输出风险等级和拦截决策 |
| F-3.5 | 工具调用审计 | P0 | 每次工具调用写入审计日志 |

### 4.4 记忆与知识管理

| 编号 | 需求 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| F-4.1 | Obsidian 双向同步 | P0 | 可读取项目背景并写入任务复盘 |
| F-4.2 | 会话状态持久化 | P0 | 每个 LangGraph 节点完成后写入 checkpoint |
| F-4.3 | 长期知识检索 | P1 | 支持从历史笔记中检索相关上下文 |
| F-4.4 | 记忆压缩 | P1 | 长任务自动生成阶段摘要，避免上下文过载 |

## 5. 非功能需求

| 编号 | 需求 | 说明 | 验收标准 |
| --- | --- | --- | --- |
| N-1 | 资源隔离 | 同一物理目录的写操作必须串行 | 并发请求不会同时修改同一仓库 |
| N-2 | 稳定性 | 单任务失败不得导致主进程退出 | 未捕获异常为 0，关键异常可恢复 |
| N-3 | 上下文控制 | 长任务需要滑动窗口或压缩 | 任务历史可追溯且不会无限膨胀 |
| N-4 | 可观测性 | 需要日志、状态、审计记录 | 可查询任务生命周期和工具调用历史 |
| N-5 | 安全性 | 高危操作默认拦截 | 未授权不执行写远端、删除、部署等动作 |
| N-6 | 可配置性 | 飞书、仓库、模型、权限规则可配置 | 配置变更无需改代码 |

## 6. 风险分级策略

| 风险等级 | 操作示例 | 默认策略 |
| --- | --- | --- |
| Low | 读取文件、查看 Git 状态、运行只读查询 | 直接执行并记录 |
| Medium | 运行测试、安装依赖、写 Obsidian 笔记 | 可执行，但需要超时和日志 |
| High | 修改代码、创建提交、推送分支、部署预览 | 根据配置决定是否授权 |
| Critical | 删除目录、强制推送、生产部署、改全局配置 | 必须人工授权 |

## 7. 系统边界与 v1.0 不做事项

- 不支持飞书语音转文字。
- 不支持通过截图让模型识别错误。
- 不提供云端安全沙箱或多租户权限隔离。
- 不开发独立 Web Dashboard。
- 不承诺完全无人值守执行生产级变更。
- 不自动处理无法确认影响范围的破坏性命令。

## 8. MVP 范围建议

### 8.1 MVP 必须交付

1. 飞书文本消息接入。
2. 飞书授权卡片回调。
3. LangGraph 基础任务流。
4. SQLite checkpoint。
5. 资源锁。
6. Shell Skill。
7. Obsidian 写入复盘。
8. 高危命令拦截。
9. 状态查询和任务取消。

### 8.2 MVP 暂缓

1. ChromaDB RAG。
2. GitHub PR 自动创建。
3. 文件夹监听触发。
4. 多模型动态路由。
5. 复杂 Dashboard。

## 9. 待澄清问题

1. 本地运行环境最终是 Mac、Windows，还是两者都要支持？
2. 飞书是个人机器人、企业自建应用，还是群机器人？
3. 高危命令白名单和黑名单由配置文件维护，还是每次动态判断？
4. Obsidian Vault 的目录结构和命名规范是什么？
5. 代码 Agent 统一使用 Claude Code，是否需要后续再引入其他 provider？
6. 任务恢复时是否允许自动继续执行写操作，还是恢复后必须先通知用户？
